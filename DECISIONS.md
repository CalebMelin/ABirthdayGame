## 2026-07-14 — Terrain generator (PLAN-02 task 1) design choices
- **Choice:** `TerrainSpec.jumps` and `.flatZones` are both optional (default to `[]`), rather than `jumps` required as PLAN-05's draft `LevelConfig` snippet shows. `JumpSpec.x` means "the ramp's base/start x", not its center or takeoff edge. Jump ramps render as a symmetric raised-cosine hump (smooth rise to a peak at the midpoint, smooth descent back to baseline) rather than an asymmetric ski-jump wedge. The Matter collision chain uses coarser segments (`TERRAIN.segmentTargetPx = 160`) than the visual heightmap (`sampleSpacingPx = 24`) specifically to stay under NORTH_STAR §8's "<100 physics bodies per level" budget — this ratio is a placeholder guess since level lengths/bike speed aren't tuned yet (PLAN-02 tasks 2/6), and should be re-checked once they are.
- **Why:** PLAN-02's task text left the exact `TerrainSpec` field shapes, jump anchor semantics, and ramp silhouette to the implementer ("Exact naming/shape is yours — keep it simple and documented"); none of this touches personal content or the north star's WHAT/WHY, only HOW.
- **How chosen:** Simplest option that stays easy/gentle/deterministic and blends with zero height/slope discontinuities (Rule 3, CLAUDE.md) — a symmetric smootherstep-based hump is trivially C1-continuous at both ends with no extra bookkeeping, and optional fields with safe defaults match the rest of the codebase's "total function, never throws on missing/malformed input" style (see save.ts).

## 2026-07-14 — Bike rig (PLAN-02 task 2) design choices
- **Choice:** (a) Gas/brake/air-control/assist are implemented as per-step angular-VELOCITY shaping (`scene.matter.setAngularVelocity` fed by pure control-law functions), not literal Matter torques — a fixed spin-up per tick IS the torque limit, and pure functions are unit-testable in Node. (b) The head fail-sensor is an `isSensor` PART of the chassis compound body (near-zero density), not a separate constrained body — rigid, zero dynamics impact, no extra constraint to leak. (c) `airborneRotation` resets at TAKEOFF, not landing, so PLAN-07 trick detection can read a finished flip's total any time after touchdown. (d) Brake wins over gas when both pedals are held (safer for a non-gamer mashing both). (e) The rear wheel keeps being driven while airborne (no reaction torque on the chassis; a spinning wheel bites on landing). (f) A crashed bike "goes limp" (input + assist ignored, physics keeps tumbling) rather than freezing, until the scene destroys/recreates it. (g) Added `bikeSpawnY()` + `BIKE_TUNING.spawnClearancePx` so the scene spawns with a small air gap — guards the terrain caveat that the physics chain can sit slightly above `heightAt(x)` in troughs. (h) `'terrain'` label became the exported `TERRAIN_BODY_LABEL` constant in terrain.ts, imported by bike.ts.
- **Why:** PLAN-02's task text specifies behaviors, not mechanisms; none of this touches personal content. The velocity-shaping approach keeps every feel-critical number a single commented `BIKE_TUNING` knob for the task-6 tuning pass, and reset-at-takeoff is the only reset timing that lets a polling scene reliably observe a completed flip.
- **How chosen:** Simplest options that keep the game EASY (assist mandatory, torque-limited gas, forgiving crash model) and testable without faking physics (Rule 3, CLAUDE.md).

## 2026-07-14 — GameScene v1 / camera (PLAN-02 tasks 3+4) design choices
- **Choice:** (a) Restart-on-fail uses full `scene.restart()` (not in-place destroy/recreate): Phaser's Matter plugin destroys the whole physics world on scene shutdown, so every body/constraint/listener is swept by construction; a SHUTDOWN backstop in the scene only calls the explicit `destroy()`s if the world somehow survived (calling them after the plugin's teardown would dereference the nulled world — verified in Phaser source). (c) The world-bottom fail line is the lowest terrain point + `FAIL.worldBottomMarginPx`. (d) The fail overlay shows for `FAIL.overlayDurationMs` (350ms) then restarts — measured fail→playable ~395ms, under the 500ms budget. (e) The hardcoded PLAN-02 test level lives in GameScene as an exported `TerrainSpec` (it's level config, not a global tunable); 15000px keeps the ground chain + bike at 93 bodies (< NORTH_STAR §8's 100) and a full-gas run at ~30s (inside 20–45s).
- **Why:** Plan text leaves camera/restart mechanisms to the implementer; none of this touches personal content. The fail copy "Oops! Go again 💛" is used byte-exact from NORTH_STAR §4.
- **How chosen:** Simplest options that are provably leak-free and keep every number a named knob (Rule 3, CLAUDE.md).

## 2026-07-14 — Two BIKE_TUNING nudges required to make the test level completable (PLAN-02 task 4)
- **Choice:** `gasSpinUpPerStep` 0.012 → 0.05 and `airSpinStepPerStep` 0.006 → 0.0025. Both carry TUNING NOTE comments in constants.ts; both must be revisited in task 6's feel-tuning pass.
- **Why:** Browser-measured, not guessed: (1) at 0.012 the per-step drive impulse balanced Matter's default body `frictionAir` (0.01) at ~180 px/s — a tenth of the intended top speed, unable to climb ramps, level uncompletable; 0.05 reaches the capped ~600 px/s in ~2–3s. Task text for the integration explicitly allows minimal tuning nudges when the level is otherwise uncompletable.
- **How chosen:** Smallest knob changes that satisfy the plan's own acceptance criterion, verified by scripted full-gas runs in real Chrome (Rule 2/3, CLAUDE.md).

## 2026-07-14 — Trick-input mechanic: mid-air pedal presses get full pitch authority; held-from-ground pedals ramp in late (PLAN-02 task 6)
- **Choice:** Airborne pitch authority is gated by input history (new pure functions in bike.ts: `nextPedalAirFresh`, `airPitchAuthority`, `isTrickAirPhase`, `nextHeldSteps`; knobs `heldPitchDelaySteps`/`heldPitchRampSteps`/`trickGroundDebounceSteps`/`trickPressBufferSteps`): a pedal press that begins mid-air (or within 0.2s before takeoff — "press buffering") gets authority 1 instantly; a pedal held since the ground gets 0 for the first 0.7s airborne, then ramps to 1 (so NORTH_STAR §4's "gas also rotates nose-up in mid-air" stays true on genuinely long airtime). At partial/zero authority the pedal result is blended with the auto-stabilization assist, so a gas-holding non-gamer gets the full assist on every hop.
- **Why:** The acceptance criteria are jointly impossible without an input-history distinction: "gas-only over the big ramp does NOT flip and does NOT crash" and "hold gas in the air off the same ramp = backflip" are the same physical input unless the game can tell a deliberate mid-air press from a cruise-hold. Browser-measured along the way: (1) a purely ballistic gas-held hop (first attempt: zero authority = no assist either) let natural takeoff spin build to ~90° and crash — hence the assist blend; (2) crest wheel-chatter on the coarse collision chords ate mid-air presses (freshness cleared by 1-3-step ground touches) — hence the 10-step ground debounce; (3) requiring the press strictly after takeoff cost ~140° of rotation to reaction latency — hence the 12-step press buffer.
- **How chosen:** Only design found that satisfies NORTH_STAR §4's control description, "a non-gamer never accidentally flips", and "flips achievable but optional" simultaneously; every piece is a commented BIKE_TUNING knob + unit-tested pure function (Rule 3, CLAUDE.md).

## 2026-07-14 — Gravity stays 1.0 (0.8 tried and rejected); flip airtime comes from ramp GEOMETRY (PLAN-02 task 6)
- **Choice:** `GRAVITY_Y` remains 1. The TEST_LEVEL trick ramp is a 336x106 raised cosine whose base/peak/end are aligned to the 168px physics collision-chain grid (nodes 10584/10752/10920), which the coarse chain renders as a clean triangle KICKER — the bike launches at the up-chord angle with real upward velocity (~0.55-1.0s air). A flat run-up zone (10000-10424) precedes it so approach speed (and flip airtime) is reproducible; the zone ends a full flatBlendPx before the ramp base because flat zones are stamped after jumps and would squash the ramp face.
- **Why:** Browser-measured, not guessed: at gravity 0.8 the ~25% longer hops chained into compounding suspension-rebound launches — one lip kicked the chassis ~90° nose-up under gas-only input and crashed a run that is rock-solid at 1.0 (gas-only survival is sacred). Meanwhile a BIG raised cosine (140x700) is NOT a kicker: climbing 140px bleeds most of the speed budget (v²/2g ≈ 210px) and the chain's flat-top chord launches the bike flat — it measured LESS airtime (27 steps) than the aligned 106px kicker (32-59 steps). Ramp height 106 is the max the 45° maxJumpSlope cap allows at width 336 without auto-widening breaking node alignment.
- **How chosen:** Measured alternatives, kept the one that leaves verified behavior untouched (Rule 2/3, CLAUDE.md).

## 2026-07-14 — Anti-endo braking: weak front brake + grounded brake stability damping (PLAN-02 task 6)
- **Choice:** The front wheel brakes at `frontBrakeFraction` (0.25) of the rear's damping, and while the brake is held on the ground the chassis' angular velocity is damped by `brakeGroundStabilization` (0.35) per step (new pure `brakingChassisAngularVelocity` — damping only, deliberately NO spring toward upright, so slope attitudes are never fought).
- **Why:** Browser-measured: a full-brake stop from top speed pitched the bike over its front axle to 108-209° nose-down (braking at the friction limit exceeds the ~0.77g geometric overturn threshold; balancing per-wheel brake strength alone could not fix it because both wheels' demands stay friction-saturated). With the damping assist, GROUNDED chassis pitch peaks at 5-6° on flat stops and 10-26° through top-speed rolling descents (envelope over 5 independent playtest runs; earlier point figures of "~10°/~38°" were contaminated — the measurement mixed in airborne hop attitudes and let a timed flat test drift into the hills, fixed in the brake-gate rework below) — no endos, no crashes. A scared non-gamer slamming the brake must never faceplant (EASY mandate).
- **How chosen:** Simplest law that provably fixes the measured failure without touching slope-following physics (Rule 3, CLAUDE.md).

## 2026-07-14 — scripts/playtest-tricks.mjs dispatches trick input page-side (PLAN-02 task 6)
- **Choice:** The committed flip playtest drives the whole recipe from a page-side monitor (window KeyboardEvent dispatch keyed to game state at ~8ms granularity) instead of CDP `keyboard.*` calls from the harness.
- **Why:** Two measured artifacts a real finger cannot produce: (1) a CDP keyup+keydown pair can land between two render frames, so the game's per-frame pedal sampling never sees the release (the "swap" silently becomes a continuous hold — no trick); (2) harness poll roundtrips add 30-40ms of jitter to timing that a human reacting to the visible ramp beats easily. The gesture itself stays human-scale (90ms release gap, state-triggered, nothing frame-perfect).
- **How chosen:** The harness must be at least as capable as a human hand for the evidence to mean anything (Rule 2, CLAUDE.md).

## 2026-07-14 — Brake-mode playtest gate measures GROUNDED pitch only; descent limit 60° (PLAN-02 task 6 review fix)
- **Choice:** `scripts/playtest-tricks.mjs` brake mode now (1) actually gates on the rolling-descent measurement (the exit condition previously checked only the flat figure — review-caught bug), (2) measures max |chassis angle| over grounded telemetry frames only, (3) keys the flat-stop test to bike POSITION (brake at x≥550, measure only x<700 frames) instead of a 3.5s timed press, and (4) uses limits of 45° flat / 60° descents. Also: playtest-drive.mjs now asserts avg FPS ≥ 55 mid-drive (the PLAN-02 "60fps" criterion previously had no scripted evidence), and playtest-out/ is gitignored.

Grounded-only + position-keyed: flat 5-6°, descents 10-34° over 6 runs. The 60° descent limit sits midway between that envelope and a real endo (90°+), so the gate can't false-fail on slope alignment yet can't miss a faceplant regression.
- **How chosen:** Measure, then set limits from the envelope — evidence before assertions (Rule 2, CLAUDE.md).

## 2026-07-15 — Touch-pedal visibility gate is `maxTouchPoints > 0` AND `(any-pointer: coarse)`, not maxTouchPoints alone (PLAN-03 task 2)
- **Choice:** `pedals.ts isTouchDevice()` shows the touch pedals only when `navigator.maxTouchPoints > 0` AND `window.matchMedia('(any-pointer: coarse)').matches`. The task text named `navigator.maxTouchPoints > 0` as the sole gate; I AND-ed a coarse-pointer media query onto it.
- **Why:** The task states TWO equally explicit requirements — "visible on touch-capable devices only (`navigator.maxTouchPoints > 0`)" AND "always hidden on pure desktop" — that conflict on Windows: browser-verified, headless system Chrome on this Windows host reports `navigator.maxTouchPoints = 10` on a PLAIN non-touch context (the OS advertises up to 10 touch contacts regardless of whether a touchscreen exists), so maxTouchPoints alone would show the pedals on a mouse-only Windows desktop. `(any-pointer: coarse)` is `false` on that same desktop and `true` on every phone/tablet/touch-2-in-1 (probed: PLAIN → coarse false, hasTouch/isMobile → coarse true), so the AND satisfies the "hidden on pure desktop" requirement while never hiding pedals on a real touch device (a phone passes BOTH terms — the safe direction). This also makes the desktop-absence acceptance check provable in `scripts/playtest-touch.mjs`, which it is not under maxTouchPoints alone on Windows.
- **How chosen:** Rule 3 judgment call resolving two conflicting explicit requirements toward the stated GOAL (hidden on pure desktop) with the smallest robust, standard idiom; no personal content involved. Verified in real Chrome across plain/touch contexts.

## 2026-07-14 — Debug overlay (PLAN-02 task 5) inlined in update()/create() instead of a private method
- **Choice:** The dev-only debug overlay's read/render logic lives directly inside `if (import.meta.env.DEV) { ... }` blocks in `GameScene.update()`/`create()`, not in a separate `private updateDebugOverlay()` method. Also: overlay text is built with plain ASCII ("deg", not "°") for pixel-font glyph safety, degrees use `Phaser.Math.RadToDeg` (no manual conversion-factor magic number), and `debugVisible` deliberately does NOT reset in `create()` (unlike `ended`/`lookaheadPx`) so the on/off toggle survives a fail-restart.
- **Why:** First attempt DID use a separate method called from inside the DEV gate; building confirmed (via `npm run build` + grepping `dist/assets/*.js` for overlay-only strings like `"resets @ takeoff"`) that Vite's minifier strips a dead `if (false) {...}` *call site* but still emits an unused class *method* as a reachable member, since it can't prove nothing else references it — so the labels/strings leaked into the production bundle. Inlining (matching main.ts's own `__gabbyGame` pattern exactly) let the whole branch, strings included, get eliminated; re-verified absent after the fix.
- **How chosen:** Simplest fix that satisfies the plan's explicit tree-shaking requirement, verified by direct inspection of the built artifact rather than assumption (Rule 2/3, CLAUDE.md).
