# CLAUDE.md — Agent Operating Instructions for "Gabby is 22!!"

You are working on a pixel-art motorcycle game that is a **birthday gift**. Quality and faithfulness to the spec matter more than speed.

## Rule 0: The North Star

**Read `NORTH_STAR.md` in full before doing anything.** Re-read the relevant section whenever you are:
- unsure what a feature should do,
- tempted to change scope, difficulty, tech stack, or any personal content,
- resolving a conflict between plan files, code, or your own judgment.

`NORTH_STAR.md` always wins. Plans in `plans/` explain HOW; the north star defines WHAT and WHY. Never edit `NORTH_STAR.md` unless Caleb explicitly asks.

## Rule 1: Work the plans in order

Execute `plans/PLAN-00-setup.md` through `plans/PLAN-11-qa-deploy.md` sequentially. Each plan has:
- **Prerequisites** — verify them; if unmet, go fix the earlier plan first.
- **Tasks** — implement in order.
- **Acceptance criteria** — ALL must pass before the plan is "done".
- **👤 CALEB STEPS** — actions only the human can do (accounts, phone testing). When you hit one, stop and give Caleb the exact instructions from the plan, expanded with anything specific to the current state of the project. Do not fake or skip these.

Track progress in `PROGRESS.md` at repo root: one line per plan (`PLAN-03 ✅ 2026-07-14 — notes`), plus any known issues.

## Rule 2: Verify before you claim

- Run `npm run build` and `npm run test` before finishing any session.
- After gameplay changes, actually play the affected level(s) in the browser (use `npm run dev`). "It compiles" is not "it works".
- Never mark a plan complete with failing tests, console errors, or an unbeatable level.

## Rule 3: Judgment calls

If something is ambiguous and the north star doesn't answer it:
1. Choose the simplest option that keeps the game EASY, CUTE, and PERSONAL.
2. Log it in `DECISIONS.md`: date, question, choice, why.
3. Don't ask Caleb unless it touches personal content (names, notes, easter eggs) — those must be exact, so ask if unsure.

## Rule 4: Things you must never do

- Change the tech stack (Phaser 3 + Matter + TypeScript + Vite, static Vercel deploy).
- Add fuel, lives, timers-that-fail-you, ads, login, backends, or difficulty spikes.
- Paraphrase, "improve", or tone down the fixed notes and easter-egg text — they are verbatim inside jokes (see NORTH_STAR §5–§6).
- Commit `node_modules`, build output, or secrets.
- Leave the game in a broken state at the end of a session — commit working checkpoints.

## Conventions

- Commits: conventional-ish, small, e.g. `feat(levels): add level 07 invisible-cars traffic system`.
- TypeScript strict mode on; no `any` unless commented why.
- All tunable gameplay numbers live in `src/systems/constants.ts` or in level configs — no magic numbers buried in scene code.
- Assets are generated by scripts in `src/art/` (run via `npm run art`) — never hand-edit PNGs in `public/assets/`; edit the generator.
- Keep every level beatable in 20–45 seconds by a mediocre player.
